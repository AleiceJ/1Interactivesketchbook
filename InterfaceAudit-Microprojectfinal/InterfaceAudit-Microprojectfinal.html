Visibility of System Status

The microproject basically does not provide any visual system status, maybe only until the end when you complete all the levels. There is also little indication of state while playing the game, because it is not paired with clear feedback or system status, the user is unsure where they are within the game, if they are doing something wrong, the user has no tell or sign that they are. I could say that the status of the rainbow is clearly defined when the rainbow is filling in, and the letters disappear from the top, the user knows they are using the right letter and the rainbow is becoming more full. The biggest weakness is the absence of visual system state. The page does not clearly indicate when a user has interacted with something or what the result of that interaction is.

Affordances & Signifiers

There are no clear signifiers within the microproject. Some may say that it is exploration based, through clicking or using the keyboard. Some elements look interactive,like the letters but are not buttons, it would be better if they were able to be clicked and use the keyboard, so the user doesn't have to guess whether an element is interactive or not. There are no clickable elements, which without properrly telling the user what to do, such as use the keyboard, the user is unsure of how to interact with the interface.

Navigation & Mapping

Navigation is expolratory. My project lacks clear navigation cues, like a menu, tabs, or sections. This layout does not strongly reflect an underlying structure. There is no hierarchy to the system, and so users don't know what to engage with first, or what exactly relates to the other in a system. Looking at this, I can say I predicted that users would just know what to do while interacting, and that made for a big lack in mapping and navigation.

Constraints & Assumptions

My projectassumes that users are comfortable exploring interfaces without guidance. What is hidden includes: The full range of interactions available The structure of the content The intended “path” through the experience What requires guessing: Which elements are interactive What each interaction will do Whether there is a correct or intended order Key assumption: The user understands that this is an exploratory interface and is willing to experiment without guidance.
Heuristic Violations
Visibility of System Status is a key violation in this microproject. The interface does not provide clear feedback about the user's actions or the system's state, making it difficult for users to understand what is happening and how their interactions are being processed. Also, recognition rather than recall is missing, the project forces the user to remember information instead of providing cues to help them recognize the correct actions.
Structural Revision Proposal
1. Establish a clear hierarchy , which could incclude labeled sections, and menus. 2. Add a navigation system like progress markeer, or menu which could help guide the user through the experience, rather than having them guess. 3. Create clearly defined intreractive elements, like the box of letters should be labeled as interactive, and the user should be able to click on the letters instead of just using the keyboard, this would make it more clear that they are interactive and how to interact with them.
